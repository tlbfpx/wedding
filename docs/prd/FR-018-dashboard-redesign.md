# FR-018: 工作台（Dashboard）管理报表重构

> **优先级**: P0（高）
> **所属模块**: 工作台
> **关联用户故事**: US-DASH-001, US-DASH-002, US-DASH-003
> **状态**: 待开发

---

## 1. 功能概述

当前工作台偏重操作层监控，缺乏管理决策所需的经营分析能力。本重构旨在将工作台升级为**管理层驾驶舱**，提供经营健康度、现金流、团队效能、风险预警、决策支撑五大维度的数据洞察，让管理层"扫一眼即可把握经营现状，深入即可找到问题根因"。

### 1.1 现状问题分析

| 现有模块 | 管理层评价 | 问题 |
|----------|-----------|------|
| 统计卡片（客户/订单/营收/活动） | 过于基础 | 缺少趋势对比、目标达成、同比环比 |
| 销售排行 | 信息有限 | 只看金额，缺少客单价、转化率、人效 |
| 转化漏斗 | 静态快照 | 缺少转化率、流失分析、来源ROI |
| 财务摘要 | 层次不足 | 未区分应收/逾期、未展示现金流、无周转分析 |
| 日程热力图 | 操作工具 | 管理层不需要看日历，需要看资源利用率 |
| 供应商排行 | 低优先级 | 供应商管理是执行层关注点 |

### 1.2 重构目标

**定位转变**：从"操作层仪表盘"升级为"管理层驾驶舱"

**核心价值**：
1. **经营健康一目了然** — 核心指标 + 趋势 + 目标达成，3秒判断经营状况
2. **问题快速定位** — 异常数据高亮 + 下钻分析，快速找到问题根因
3. **决策有据可依** — 客户来源ROI、服务类型占比、供应商性价比，支撑资源分配决策
4. **风险主动预警** — 逾期应收、即将到期活动、异常指标，提前介入处理

## 2. 用户角色与权限

### 2.1 角色定义

| 角色 | 关注点 | 查看权限 |
|------|--------|----------|
| **老板/总经理** | 全局经营健康、目标达成、现金流、重大风险 | 全局数据 |
| **销售总监** | 销售业绩、转化效率、团队效能、客户来源ROI | 全局 + 团队对比 |
| **策划总监** | 排期负载、资源利用率、活动交付质量 | 排期相关数据 |
| **财务总监** | 应收/现金流、付款健康度、成本结构 | 财务相关数据 |

### 2.2 权限矩阵

| 模块 | 老板 | 销售总监 | 策划总监 | 财务总监 | 销售主管 | 销售 |
|------|------|----------|----------|----------|----------|------|
| 经营健康度 | 全公司 | 全公司 + 团队 | 只读 | 只读 | 本团队 | 本人 |
| 现金流分析 | 全公司 | 只读 | 只读 | 全公司 | 只读 | 无 |
| 团队效能 | 全公司 | 全公司 | 只读 | 只读 | 本团队 | 本人 |
| 风险预警 | 全公司 | 全公司 | 本团队相关 | 全公司 | 本团队 | 本人 |
| 决策支撑 | 全公司 | 全公司 | - | 全公司 | 本团队 | - |

### 2.3 权限标识（RBAC）

- `dashboard:read` — 查看工作台基础数据
- `dashboard:read_all` — 查看全局数据（老板/总监级）
- `dashboard:read_team` — 查看团队数据（主管级）
- `finance:read` — 查看财务数据（现金流/应收）

## 3. 报表模块设计

### 3.1 模块一：经营健康度（核心卡片区）

#### 设计目的
管理层打开工作台的第一眼，立即了解公司整体经营状况和健康程度。

#### 核心指标

| 指标 | 展示内容 | 趋势/对比 | 目标达成 |
|------|----------|-----------|----------|
| **营收** | 本期营收（支持月/季/年切换） | 环比增长率、同比增减额 | 目标完成率（需支持配置目标） |
| **订单** | 本期订单数 | 环比变化 | - |
| **客单价** | 本期平均订单金额 | 环比变化 | - |
| **签约率** | 意向→签约转化率 | 环比变化 | - |
| **毛利** | 营收 - 已付供应商款 | 环比变化 | 目标完成率 |

#### 前端交互设计

```
┌─────────────────────────────────────────────────────────────────────┐
│  经营健康度                                  [本月 ▼]  [对比上月]    │
├─────────────────────────────────────────────────────────────────────┤
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐ │
│  │  营收    │  │  订单    │  │  客单价  │  │  签约率  │  │  毛利    │ │
│  │ ¥128.5万 │  │   42单   │  │ ¥3.06万  │  │  68%    │  │ ¥45.2万  │ │
│  │ ▲ 12.5%  │  │ ▲ 8.3%   │  │ ▼ 3.2%   │  │ ▲ 5.1%   │  │ ▲ 15.8%  │ │
│  │ 85%达成  │  │          │  │          │  │          │  │ 92%达成  │ │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘  └──────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

**交互说明**：
- 支持"月/季/年"切换
- 环比/同比可选（默认环比上月）
- 趋势箭头颜色：绿色增长、红色下降（可配置）
- 目标达成：进度条样式，低于80%黄色预警，低于60%红色告警
- 点击卡片可下钻到明细页面（如营收→营收明细报表）

#### 数据来源

| 字段 | 数据来源 | 计算逻辑 |
|------|----------|----------|
| 营收 | Order.total_amount | SUM(total_amount) WHERE created_at IN [本期] AND status != 'cancelled' |
| 订单数 | Order.id | COUNT(id) WHERE created_at IN [本期] AND status != 'cancelled' |
| 客单价 | 营收 / 订单数 | 营收 / 订单数 |
| 签约率 | Customer.status | COUNT(signed) / COUNT(intention) |
| 毛利 | Order - SupplierPayment | 营收 - SUM(供应商已付金额) |

### 3.2 模块二：现金流与应收（财务区）

#### 设计目的
管理层最关心的财务健康：钱进来多少、还没回来多少、是否逾期、资金周转如何。

#### 核心指标

| 指标 | 展示内容 | 风险阈值 |
|------|----------|----------|
| **现金流入** | 本期实收金额（按付款方式分布） | - |
| **应收余额** | 未收回款项总额 | > 50万预警 |
| **逾期应收** | 超过约定期限未收款项 | > 10万告警 |
| **应收周转天数** | 平均回款周期 | > 45天预警 |
| **付款健康度** | 本期应付供应商、已付、未付 | - |

#### 数据来源

| 字段 | 数据来源 | 计算逻辑 |
|------|----------|----------|
| 现金流入 | Payment.amount | SUM(amount) WHERE status='confirmed' AND paid_at IN [本期] |
| 应收余额 | Order.total_amount - Order.paid_amount | SUM(未付金额) WHERE status IN ['signed', 'executing'] |
| 逾期应收 | Receivable.due_date | SUM(未付金额) WHERE due_date < 今日 |
| 应收周转天数 | Receivable + Payment | (平均应收余额 / 本期营收) × 本期天数 |
| 付款方式分布 | Payment.method | GROUP BY method, SUM(amount) |

### 3.4 模块四：风险预警

#### 设计目的
主动发现经营风险，提前介入处理，避免小问题变成大损失。

#### 预警类型与规则

| 预警类型 | 触发条件 | 级别 | 处理建议 |
|----------|----------|------|----------|
| **逾期应收** | 单笔逾期 > 5万 或 累计逾期 > 10万 | 高 | 立即联系客户催收 |
| **即将到期活动** | 3天内未确认的活动 | 中 | 确认准备情况 |
| **长期未跟进** | 意向客户 > 7天未跟进 | 中 | 安排跟进 |
| **高风险客户** | 预算低于平均客单价30% + 首次咨询 | 中 | 谨慎投入资源 |
| **供应商风险** | 供应商评分 < 3.5 + 本月有排期 | 中 | 准备备选方案 |
| **目标告急** | 营收达成率 < 60% 且本月只剩10天 | 高 | 调整策略或降低预期 |

#### 前端交互设计

```

│  风险预警                                              [自动刷新:5分钟] │
├─────────────────────────────────────────────────────────────────────┤

│  │    操作：[查看营收明细] [调整目标]                            │   │
│  ├───────────────────────────────────────────────────────────────┤   │
│  │ 🔴 逾期应收：客户王五的婚礼订单(¥45,000)已逾期12天            │   │

│  ├───────────────────────────────────────────────────────────────┤   │
│  │ 🟡 长期未跟进：客户陈九(意向状态)已12天未跟进                 │   │
│  │    负责人：吴十 | 操作：[查看客户] [分配跟进]                 │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                       │
│  [查看全部预警]  [配置预警规则]  [导出预警报告]                       │
└─────────────────────────────────────────────────────────────────────┘
```

**交互说明**：
- 预警卡片按级别分组，高风险优先展示
- 每条预警提供明确的操作入口
- 支持标记已处理（不消失，记录在历史中）
- 自动刷新（可配置间隔）

#### 数据来源

| 预警类型 | 数据来源 | 计算逻辑 |
|----------|----------|----------|
| 逾期应收 | Order + Receivable | WHERE due_date < TODAY - paid_amount < total_amount |
| 即将到期活动 | Event | WHERE date BETWEEN TODAY AND TODAY+3 AND status != 'confirmed' |
| 长期未跟进 | Customer + FollowUp | WHERE status='intention' AND last_follow_at < TODAY-7 |
| 高风险客户 | Customer.budget_range | 解析预算下限，对比平均客单价 |
| 供应商风险 | Supplier.rating + Event | WHERE rating < 3.5 AND EXISTS(event in current month) |
| 目标告急 | RevenueTarget + Order | 目标达成率计算 + 剩余天数判断 |

### 3.5 模块五：决策支撑（ROI与成本分析）

#### 设计目的
为管理层提供资源分配决策依据：哪个获客渠道最有效、哪些服务最赚钱、哪些供应商性价比最高。

#### 核心分析

| 分析维度 | 展示内容 | 决策价值 |
|----------|----------|----------|
| **客户来源ROI** | 各渠道获客成本 vs 产出 | 决定投放重点 |
| **服务类型收入占比** | 各服务类型（策划/主持/摄影等）收入 | 决定产品重点 |
| **供应商性价比** | 供应商价格 vs 评分 vs 合作频次 | 决定合作策略 |
| **客户预算分布** | 客户预算区间 vs 成交率 | 决定定价策略 |

4. **增量刷新**：自动刷新时仅更新数值，不重绘整个页面

|----------|-----------|--------|----------|----------|----------|

| 销售主管 | 全公司 + 团队对比 | 本人团队 | 本人团队 | 本人团队 | 本人团队 |
| 销售总监 | 全公司 + 团队对比 | 只读 | 全公司 | 全公司 | 全公司 |
| 销售 | 本人数据 | 无 | 本人数据 | 本人数据 | 无 |
| 策划总监 | 只读 | 无 | 只读 | 只读 | - |
| 策划师 | 只读 | 无 | 本人数据 | 本人数据 | - |
| 财务总监 | 只读 | 全公司 | 只读 | 全公司 | 全公司 |

### 6.2 权限实现

后端通过 `require_permission` 中间件控制：

```python
# 示例：团队效能数据根据用户角色过滤
@router.get("/team-efficiency")
async def get_team_efficiency(
    period: str = Query("month"),
    team: Optional[str] = Query(None),
    db: AsyncSession = Depends(get_db),
    ctx: dict = Depends(require_permission("dashboard", "read")),
):
    user = ctx["user"]
    scope = ctx.get("scope", "own")  # all, team, own

    if scope == "own":
        # 只返回本人数据
        query = query.where(User.id == user.id)
    elif scope == "team":
        # 返回本团队数据
        query = query.where(User.team == user.team)
    # else: all，无过滤
```
