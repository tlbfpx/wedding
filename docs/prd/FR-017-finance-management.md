# FR-017: 财务管理模块

> **优先级**: P0（高）
> **所属模块**: 财务管理
> **关联用户故事**: US-FIN-001, US-FIN-002, US-FIN-003, US-FIN-004
> **状态**: 待开发

---

## 1. 功能概述

财务管理模块为婚庆管理系统提供完整的财务管控能力，将分散在订单、审批、报表中的财务功能整合到独立模块中，实现应收账款跟踪、收款核销、退款管理、收支明细记录、开票管理和财务对账等核心财务功能，确保财务数据的准确性、可追溯性和合规性。

**核心价值**:
- 统一财务数据入口，避免信息分散
- 自动跟踪应收账款，减少逾期风险
- 规范退款流程，降低财务风险
- 完整收支记录，支持财务对账
- 开票状态跟踪，确保税务合规

## 2. 用户角色与权限

| 角色 | 财务模块权限范围 | 说明 |
|------|-----------------|------|
| 管理员 | 全部权限 | 财务配置、对账、开票、退款审批 |
| 销售主管 | 收款登记、收支明细查看 | 本团队收款管理、数据查看 |
| 销售 | 收款登记（关联自己订单） | 记录自己负责订单的收款 |

| 设计主管/设计师 | 无权限 | - |

**权限标识**（RBAC）:
- `finance:read` - 查看财务数据（应收、收支明细、对账报表）
- `finance:write` - 财务操作（收款登记、退款申请、开票申请）
- `finance:approve` - 财务审批（退款审批、对账确认）
- `finance:export` - 财务报表导出

## 3. 功能明细

### 3.1 应收账款管理

**功能点列表**:
- 基于订单自动生成应收记录
- 应收状态跟踪（未收款/部分收款/已收款/已逾期）
- 逾期预警（超过约定期限未收款）
- 应收账款列表与筛选

**页面/入口**:
- `/finance/receivables` - 应收账款管理页面
- 订单详情页内嵌"应收状态"卡片

**核心业务流程**:
1. 订单创建后自动生成应收记录（状态：未收款）
2. 订单签约后，根据合同约定计算应收到期日（默认签约后30天）
3. 收款登记后自动核销应收，更新应收状态
4. 部分收款时更新应收为"部分收款"状态
5. 到期日超过且未全额收款时标记为"已逾期"

**后端 API**:
- `GET /api/v1/finance/receivables` - 应收账款列表（支持按状态、日期范围、销售筛选）
- `GET /api/v1/finance/receivables/:id` - 应收详情
- `GET /api/v1/finance/receivables/overdue` - 逾期应收列表

---

### 3.2 收款登记与核销

**功能点列表**:
- 收款登记（关联订单、金额、付款方式、收款日期）
- 支持部分收款（订单可分多次收款）
- 自动核销应收账款
- 收款记录查询与修改（未审核状态可修改）
- 收款确认（审核机制，可选）

**页面/入口**:
- `/finance/payments` - 收款管理页面
- `/finance/payments/create` - 收款登记页面
- 订单详情页"收款记录"tab

**核心业务流程**:
1. 财务人员/销售填写收款信息（订单、金额、付款方式）
2. 系统校验：收款金额 + 已收金额 <= 订单总额
3. 收款记录保存后自动更新：
   - 订单的 `paid_amount`
   - 对应收记录进行核销
   - 计算应收状态
4. 支持收款后修改（需审核权限），修改后重新计算
5. 收款超额时拦截并提示

**后端 API**:
- `POST /api/v1/finance/payments` - 登记收款
- `GET /api/v1/finance/payments` - 收款记录列表（支持按订单、付款方式、日期筛选）
- `PUT /api/v1/finance/payments/:id` - 修改收款记录（需审核权限）
- `DELETE /api/v1/finance/payments/:id` - 删除收款记录（需审核权限）

**业务规则**:
1. 收款金额不可为负数
2. 收款后总额不超过订单总额
3. 付款方式：现金/转账/微信/支付宝/刷卡
4. 收款记录删除后需重新计算订单已付金额

---

### 3.3 退款管理

**功能点列表**:
- 退款申请（关联订单、退款金额、退款原因）
- 退款审批流程（复用现有审批模块）
- 退款记录（审批通过后生成）
- 退款状态跟踪（待审批/已批准/已拒绝/已退款）
- 退款对订单已付金额的影响

**页面/入口**:
- `/finance/refunds` - 退款管理页面
- `/finance/refunds/create` - 退款申请页面
- 审批管理页面查看退款审批

**核心业务流程**:
1. 财务人员/销售发起退款申请
2. 系统校验：退款金额 <= 订单已付金额
3. 创建退款记录（状态：待审批），同时创建审批记录
4. 审批通过后：
   - 退款记录状态变为"已批准"
   - 订单 `paid_amount` 扣减退款金额
   - 应收记录重新计算
5. 审批驳回后退款记录状态变为"已拒绝"
6. 实际退款完成后标记为"已退款"

**后端 API**:
- `POST /api/v1/finance/refunds` - 申请退款
- `GET /api/v1/finance/refunds` - 退款记录列表
- `GET /api/v1/finance/refunds/:id` - 退款详情
- `PUT /api/v1/finance/refunds/:id/status` - 更新退款状态（标记已退款）

**与现有审批集成**:
- 退款申请自动创建 `Approval` 记录（type=refund）
- 审批通过后回调更新退款记录状态
- 审批驳回后退款记录状态同步更新

---

### 3.4 收支明细

**功能点列表**:
- 统一记录所有收入（客户付款）和支出
- 支出分类：供应商付款、其他支出（人工、场地、物料等）
- 收支明细列表（时间倒序）
- 支持多维度筛选（类型、分类、日期、关联订单/供应商）
- 收支汇总统计

**页面/入口**:
- `/finance/transactions` - 收支明细页面

**核心业务流程**:
1. 收入类目：
   - 客户付款：与收款记录关联，自动生成
2. 支出类目：
   - 供应商付款：关联订单项和供应商
   - 其他支出：手动登记（类型、金额、日期、备注）
3. 收支记录自动关联创建人/创建时间
4. 支持导出 Excel 用于财务分析

**后端 API**:
- `GET /api/v1/finance/transactions` - 收支明细列表（支持筛选、分页）
- `POST /api/v1/finance/transactions` - 手动登记支出
- `PUT /api/v1/finance/transactions/:id` - 修改支出记录
- `GET /api/v1/finance/transactions/summary` - 收支汇总统计

**业务规则**:
1. 收入记录自动生成，不可手动创建
2. 支出记录可手动创建和修改
3. 收支记录删除需管理员权限
4. 删除收入记录需同时删除关联的收款记录

---

### 3.5 开票管理

**功能点列表**:
- 开票申请（关联订单、发票类型、金额、抬头信息）
- 开票记录列表与状态跟踪（待开票/开票中/已开票/已作废）
- 发票信息管理（发票号码、开票日期、PDF 附件）
- 开票审批（超过一定金额需审批，可配置）
- 开票状态与订单状态联动

**页面/入口**:
- `/finance/invoices` - 开票管理页面
- `/finance/invoices/create` - 开票申请页面
- 订单详情页"开票记录"tab

**核心业务流程**:
1. 销售发起开票申请（选择订单、填写开票信息）
2. 系统校验：开票金额 <= 订单总额
3. 创建开票记录（状态：待开票）
4. 若开票金额超过阈值（可配置，默认 50000 元），需审批
5. 财务人员处理开票，上传发票 PDF，更新状态为"已开票"
6. 支持发票作废（需审批）

**后端 API**:
- `POST /api/v1/finance/invoices` - 申请开票
- `GET /api/v1/finance/invoices` - 开票记录列表
- `GET /api/v1/finance/invoices/:id` - 开票详情
- `PUT /api/v1/finance/invoices/:id` - 更新开票状态/上传发票
- `DELETE /api/v1/finance/invoices/:id` - 作废发票

**业务规则**:
1. 发票类型：增值税普通发票/增值税专用发票
2. 开票金额默认为订单总额，可手动调整（不可超过）
3. 发票号码唯一性校验
4. 已开票记录不可修改开票金额

---

### 3.6 财务对账

**功能点列表**:
- 按月/季度生成对账报表
- 收入对账：核对收款记录与订单应收的一致性
- 支出对账：核对供应商付款与订单项成本的一致性
- 对账差异报告
- 对账确认与归档

**页面/入口**:
- `/finance/reconciliation` - 财务对账页面

**核心业务流程**:
1. 选择对账周期（月份或季度）
2. 系统生成对账报表：
   - 收入：应收总额 vs 实收总额，差异明细
   - 支出：应付总额 vs 实付总额，差异明细
3. 展示差异详情（哪些订单未全额收款/付款）
4. 财务人员确认对账结果
5. 对账记录归档，支持历史查询

**后端 API**:
- `GET /api/v1/finance/reconciliation/report` - 生成对账报表
- `POST /api/v1/finance/reconciliation/confirm` - 确认对账
- `GET /api/v1/finance/reconciliation/history` - 对账历史记录

**业务规则**:
1. 对账周期按月或按季度
2. 对账基于收款/付款的实际发生日期
3. 对账确认后生成快照，历史数据不受后续变更影响
4. 仅管理员可确认对账

## 4. 数据实体

### 4.1 新增实体

| 实体 | 说明 | 关键字段 |
|------|------|----------|
| **Receivable** | 应收账款 | order_id(FK), total_amount, received_amount, status, due_date |
| **FinancePayment** | 收款记录（扩展 Payment） | order_id(FK), amount, method, paid_at, status, created_by(FK) |
| **Refund** | 退款记录 | order_id(FK), amount, reason, status, approval_id(FK), approved_by, refunded_at |
| **Transaction** | 收支明细 | type(收入/支出), category, amount, order_id(FK), supplier_id(FK), date, note |
| **Invoice** | 开票记录 | order_id(FK), invoice_type, amount, title, tax_no, status, invoice_no, pdf_url, issued_at |

### 4.2 枚举定义

**应收状态 (ReceivableStatus)**:
- `unpaid` - 未收款
- `partial` - 部分收款
- `paid` - 已收款
- `overdue` - 已逾期

**退款状态 (RefundStatus)**:
- `pending_approval` - 待审批
- `approved` - 已批准
- `rejected` - 已拒绝
- `refunded` - 已退款

**收支类型 (TransactionType)**:
- `income` - 收入
- `expense` - 支出

**支出分类 (ExpenseCategory)**:
- `supplier_payment` - 供应商付款
- `labor` - 人工费用
- `venue` - 场地费用
- `material` - 物料费用
- `other` - 其他

**发票类型 (InvoiceType)**:
- `normal` - 增值税普通发票
- `special` - 增值税专用发票

**开票状态 (InvoiceStatus)**:
- `pending` - 待开票
- `processing` - 开票中
- `issued` - 已开票
- `voided` - 已作废

### 4.3 实体关系

```
Order (1) ──── (1) Receivable
    │
    ├── (N) FinancePayment ──→ Transaction (收入，自动生成)
    │
    ├── (N) Refund ──→ Approval (审批)
    │
    └── (N) Invoice

OrderItem (N) ──→ Transaction (支出-供应商付款)

Supplier (1) ──── (N) Transaction (支出关联)
```

## 5. 与现有模块的集成点

### 5.1 与订单管理集成

| 集成点 | 说明 |
|--------|------|
| 订单创建 | 自动生成 Receivable 记录（total_amount = order.total_amount） |

| 集成点 | 说明 |

## 6. 非功能需求

### 6.1 性能要求

- 应收列表查询响应时间 < 500ms（1000 条记录）
- 收支明细列表查询响应时间 < 500ms（1000 条记录）
- 对账报表生成时间 < 3s（月度数据）

### 6.2 安全要求

- 财务数据权限严格控制，普通用户只能看自己关联的数据
- 财务操作（收款、退款、开票）记录操作日志
- 敏感数据（客户抬头、税号）脱敏展示

### 6.3 数据一致性

- 收款/退款操作需事务保护
- 应收计算需保证与收款记录一致
- 对账快照需独立存储，不受后续数据变更影响

## 7. 验收标准

| 编号 | 验收条件 | 验证方式 |
|------|----------|----------|
| AC-001 | 订单创建后自动生成应收记录 | 创建订单并检查应收列表 |
| AC-002 | 收款登记后自动核销应收，状态正确更新 | 登记收款并验证应收状态 |
| AC-003 | 部分收款正确计算应收状态为"部分收款" | 登记部分收款并验证 |
| AC-004 | 逾期应收正确标记并展示 | 修改到期日为过去日期并验证 |
| AC-005 | 退款申请创建审批并正确处理 | 申请退款、审批通过并验证订单金额 |
| AC-006 | 收支明细正确关联收入和支出 | 登记收款和支出，验证列表展示 |
| AC-007 | 开票申请流程完整，状态流转正确 | 创建开票申请、上传发票、验证状态 |
| AC-008 | 对账报表准确统计应收和实收差异 | 生成月度对账报表并核对数据 |
| AC-009 | 权限控制生效，非授权用户无法操作 | 用不同角色账号验证权限 |
| AC-010 | 现有付款记录正确迁移到新模块 | 数据迁移脚本执行并验证 |
